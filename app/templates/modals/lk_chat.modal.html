Case Study

Summary:

Letterkenny Chat is a web application that provides a place for fans of the Canadian sitcom hit, “Letterkenny,” produced by CraveTV, to interact. More importantly, this project was an opportunity to expand my budding Angular skill set, as well as creating an application that communicates with a backend database; in this case, Firebase.

Explanation:

The goals of the project included:

  • Use of Angularjs architecture
  • Use of Nodejs for local development server and dependency package manager
  • Use of Angularjs and Vanilla Javascript to add functionality for users to view and send messages, add new chat rooms and choose a username
  • Use of Firebase database to store and reference user messages and chat room objects
  • Use of Cookies to store local username and display as data included in user message objects
  • Use of UI-Bootstrap to create modals for the user interface

Problems & Solutions:

Login and Authentication

I decided to create a true login feature, which meant I needed to consider authentication. Looking at the Firebase options, which include using it as a RESTful API, dealing with the SDK, acquiring OAuth or JWT tokens or API keys and the plethora of other options left my head spinning. Stepping back, I opted to use the AngularFire library and create the needed authentication, as well as creating an automatic registration process that would store and reference the user object within the database as needed.

Architecture

This project allowed me to learn a great deal about ui-router, directives and logic location.

Wanting to potentially populate the same dropdown menu with a login or registration form led me to using a directive to reference either template, which was then all nested in the dropdown menu. However, I disliked users having to use tab-type buttons to switch the views. This made me rethink the need for separate registration and I decided to simplify the process with an automatic registration, allowing the user a more streamlined entry and experience. With my newfound directive familiarity, I went on to use this technique to encapsulate the login template and chat room message panel as inserted elements within the main html.

My explorations with ui-router to play with state and ui-sref versus ui-view options, led me to learn a great deal about partial templates, as well as the structure of traditional states versus nested views.

Finally, I really just got more comfortable with Angular, mapping out the architecture of specific features as well as the overall program on a white board and within coggle, an online visual mapping tool. One of my biggest takeaways was the cementing of the MVC structure within my mind - starting to truly understand things like - controllers are there to communicate and link the view (browser rendered content) to the model (templates) and services are meant to provide the logic and will always return an object containing your controller referenced methods.

Design

While I did do some best practices for this project, including wireframing the initial pages/concept, I did not take into account responsive design and this caused considerable delay in the polishing phase of pre-deployment, including a total redesign of the chat room and message interface. Use of techniques including CSS’s flexbox and viewport units was revelatory.
Lesson learned = design responsively from the get go!

Results:

Letterkenny Chat successfully provides an application for fans to chat about their favorite show and its characters. Users can login/logout, create a new chatroom, select an existing room as well as read and send messages. Testing revealed issues with username data accessibility, authentication status and responsive design, allowing for solutions prior to deployment.

This application has lots of room for new features in the future, including user profiles, a live voting widget for fan favorite quotes and a facebook type page for sharing farts, based on one of the series’ most popular bits, “Fartbook.”

Conclusion:

This project has provided me with a deeper understanding of Angularjs, which will allow me to make use of techniques and best practices I had previously been unfamiliar with; providing a faster development and deployment on future applications.

While this was my first foray into authentication and was therefore very simple, it did open my eyes to the variety of techniques and technology available. I also gained a new understanding of the need for user security and common pain points like token renewal/timeout that I am likely to encounter throughout my career.

Finally, while CSS will never be a quick process, integrating responsive design from the start of a project is a huge timesaver and something I will incorporate into my process moving forward. In the age of smart devices of all sizes and our culture’s growing demand for content on the go, responsive design is not optional.
